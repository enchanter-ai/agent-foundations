# `.foundations-lock` — Schema

The `.foundations-lock` file is the resolved-state snapshot generated by
`scripts/bootstrap.sh` in every `@enchanter-ai` sibling plugin. It pairs with
`.foundations-versions` (the manifest of intent — pinned package specs) the
way `package-lock.json` pairs with `package.json`.

Per the s2.0 4-layer recommendation:

- **Layer 1 (Bootstrap)** writes this file.
- **Layer 2 (SessionStart hook)** reads it and compares to current state; fail-loud on drift.
- **Layer 3 (CI verify)** runs `./scripts/bootstrap.sh --verify` which is hash-coverage all-or-nothing against this file.

## Contract

- **Generated, not edited.** Every line is auto-emitted by `bootstrap.sh`.
- **Committed to the plugin repo.** Like `package-lock.json`, the lock is a tracked artifact — the sibling's identity includes the resolved foundations state.
- **All-or-nothing.** A `.foundations-lock` that is missing entries for any
  `@../enchanter-foundations/...` reference in `CLAUDE.md` is an error, not a degraded state. Bootstrap aborts; CI fails.

## Shape

```yaml
# .foundations-lock — auto-generated by scripts/bootstrap.sh
# Do not edit by hand. Run ./scripts/bootstrap.sh to refresh.

packages:

    version: v0.6.0

    tag_commit: bbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbb
  web:
    version: v0.6.0

  orchestration:
    version: v0.6.0
    tag_commit: dddddddddddddddddddddddddddddddddddddddd

    sha1: 1234567890abcdef1234567890abcdef12345678
  - path: packages/core/conduct/context.md
    sha1: 234567890abcdef1234567890abcdef123456789a
  # ... one entry per @../enchanter-foundations/... reference in CLAUDE.md
```

## Field-by-field

| Field | Type | Source | Used by |
|---|---|---|---|
| `foundations_commit` | git SHA (40 char hex) | `git -C ../enchanter-foundations rev-parse HEAD` at bootstrap time | Hook + CI compare against current foundations HEAD; mismatch is drift |
| `resolved_at` | ISO 8601 UTC | bootstrap timestamp | Diagnostic only; not compared |
| `packages.<name>.version` | string | parsed from `.foundations-versions` | Diagnostic only |
| `packages.<name>.tag_commit` | git SHA | `git -C ../enchanter-foundations rev-list -n 1 refs/tags/enchanter-<name>--v<version>` | Hook + CI compare against current tag resolution; mismatch = pin moved without re-bootstrap |
| `conduct_files[].path` | string | extracted from `@../enchanter-foundations/...` references in `CLAUDE.md` | Hook + CI walk this list; missing entry = under-coverage error |
| `conduct_files[].sha1` | hex (40 char) | `sha1sum` of the resolved file at bootstrap time | Hook + CI re-hash and compare; mismatch = file modified post-bootstrap (manual edit or upstream change without re-bootstrap) |

## Allowed mutations

Only `scripts/bootstrap.sh` mutates this file. Two trigger paths:

1. **Explicit:** developer runs `./scripts/bootstrap.sh` after updating `.foundations-versions`. Lock is rewritten with new resolved state.
2. **CI sanity:** CI runs `./scripts/bootstrap.sh` (no `--verify`) and asserts the resulting lock matches the committed lock byte-for-byte. If CI's regenerated lock differs from what's checked in, the PR is failed with the message `".foundations-lock is out of sync with .foundations-versions"`.

Hand-editing this file, or any tool other than `bootstrap.sh` writing it, is a contract violation.

## Failure-mode mapping

| Drift type | Detected by | Output |
|---|---|---|
| Lock missing | Hook + `--verify` | `"foundations not bootstrapped — run ./scripts/bootstrap.sh"` |
| Foundations sibling missing | Hook + `--verify` | `"foundations sibling missing — run ./scripts/bootstrap.sh"` |
| `foundations_commit` ≠ current HEAD | Hook + `--verify` | `"foundations drift: lock says <X>, checkout is <Y> — run ./scripts/bootstrap.sh to re-resolve"` |
| `packages.<pkg>.tag_commit` ≠ resolved tag | Hook + `--verify` | `"package <pkg>: lock <X>, checkout <Y> — run ./scripts/bootstrap.sh"` |
| `conduct_files[].sha1` ≠ live sha1 | Hook + `--verify` | `"conduct file <path> modified since bootstrap — re-bootstrap or revert"` |
| Conduct file referenced in `CLAUDE.md` but absent from lock | bootstrap (refuses to write) | `"import resolves to missing file: @../enchanter-foundations/<path>"` |

## Versioning

This schema is v1. Backward-incompatible changes (rename a top-level field, change `sha1` to `sha256`, add a required field) require a `lock_version: 2` line at the top and a coordinated bump across `bootstrap.sh`, the SessionStart hook, and CI. Additive optional fields do not require a version bump but should be documented here first.

## Cross-vendor analogues

Lock-file pattern is well-precedented:

- **`package-lock.json`** (npm) — same role: manifest intent + resolved state, fail loud on `npm ci` if they disagree.
- **`Cargo.lock`** (Rust) — same: lock is the source of truth for actual versions installed; manifest is the constraint.
- **`go.sum`** (Go) — hash-coverage analogue. `go mod verify` mismatch produces a `SECURITY ERROR` and non-zero exit.
- **`requirements.txt`** with `--require-hashes` (pip) — partial-coverage is an error per pip docs.

`.foundations-lock` is the @enchanter-ai analogue. It exists because @-imports without it fail silently (see Brief Pattern A.3, Git submodules).
